# 核验方法

## 一、考生基线核验

固定基线必须同时满足以下条件：

- 用户提供的成绩截图总分为 515。
- 用户提供的各科成绩相加为 515。
- 湖北官方一分一段表中，515 分累计人数为 91723。
- static section JSON 中，515 分位次区间为 `90895-91723`，累计人数为 `91723`。
- static section JSON 中，2026 年 515 分对应 2025 等位分 494、2024 等位分 497、2023 等位分 481。
- 官方 2026 控制线图片中，首选物理本科线为 435，特殊招生线为 529。

运行：

```bash
python3 scripts/verify_baseline.py
```

## 二、历年投档线核验

2025 投档线保留三层数据：

1. 湖北省教育厅官方原始图片。
2. 本地 OCR 全文。
3. 便于筛选的解析 CSV。

OCR 只是派生数据。任何从照片或图片识别出的院校代码、专业组、专业代号、计划数、学费和备注，进入候选表前都必须回看原图人工确认。

2024 和 2023 投档线保留官方 PDF、PDF 文本和解析 CSV。

最终志愿决策不能只依赖 CSV。每个入选院校专业组都必须回看对应官方原件，确认：

- 院校专业组代号
- 院校专业组名称
- 再选科目要求
- 投档最低分
- 备注，如中外合作、预科、护理类、国家专项、地方专项、民族班、定向就业等

## 三、2026 计划核验

历史投档线不能替代 2026 招生计划。每个最终候选都要核验：

- 2026 湖北招生计划或官方志愿填报系统。
- 学校 2026 本科招生章程。
- 专业组代码和专业组成是否仍然一致。
- 计划人数是否明显变化。
- 再选科目要求是否匹配“化学、地理”。
- 学费、校区、合作办学、体检限制、单科要求和备注。
- 如果 2026 计划来自第 19 期照片 OCR，必须同时核验照片原件页码、院校专业组完整范围和 OCR 输出哈希。

任何没有完成“官方计划 + 历史投档线 + 招生章程”核验的专业组，都必须保留 `needs_verification` 状态，不能进入最终表。

当前 2026 招生计划获取状态见 `docs/2026_ADMISSION_PLAN_ACQUISITION.md`。在完整计划导入前，所有院校专业组只能停留在候选池或待核验状态。

第 19 期 OCR 明细的状态流必须按以下顺序升级：

```text
OCR 初稿 / 最终可用=false
-> 原 PDF 页人工复核
-> 湖北官方平台或志愿系统核验
-> 高校官网或招生章程交叉核验
-> 家庭接受度和调剂判断
-> 三年投档稳定性判断
-> 可进入冲稳保排序
```

每次升级都要留下可追溯记录：

- 来源期号、来源 PDF SHA256、来源页码、院校代码、院校专业组代码。
- 核验日期和核验字段：专业代号、专业名称、计划数、学费、选科、校区、备注、限制条件。
- 异常说明：OCR 错字、跨页、组号变化、官网与省招办不一致、字段缺失或疑似误识别。
- 私有证据路径：截图、PDF 页图、志愿系统截图只放在 `private/`，不提交 Git。

公开 CSV 里的 `机器初判`、`综合风险等级`、`硬风险类型` 只用于安排复核顺序。命中不等于可报，未命中不等于不存在，OCR 字段不等于最终事实。

候选池页面复核包的使用方式：

- 公开元数据见 `data/working/issue19-candidate-review-page-packet.csv` 和 `data/working/issue19-candidate-review-group-page-map.csv`。
- 本地私有页图和页面 OCR 文本只用于人工回看原页，不提交 Git。
- 人工核页时要同时看候选专业组和同校相邻专业组，防止跨页、漏组、组号变化或 OCR 切组错误。
- 页面复核包生成不代表已完成核验；只有逐字段确认后，才能进入候选池 V2。

完整性审计队列的使用方式：

- `data/working/issue19-candidate-page-code-audit.csv` 用来核对候选组号在页面 OCR 和全量结构化表中的命中是否一致。
- `页面有组号但结构化未拆出` 必须优先回看页图，确认是否为自动切组漏拆；当前 `C10704` 属于这一类。
- `复核页未见候选组号且结构化未命中` 不能直接删除，需要核查 2026 组号变化、历史投档线旧组号或候选池 OCR 噪声；当前 `C10702`、`K15123` 属于这一类。
- `data/working/issue19-ocr-structure-anomaly-queue.csv` 用来定位专业明细中的串校、串组、页眉串入、专业代号异常、计划数/学费错位和低置信度行。命中异常的专业行，在人工复核前不得用于判断专业组能否服从调剂。

候选 V2 逐专业明细种子的使用方式：

- `data/working/issue19-candidate-v2-group-review-seed.csv` 只负责定位候选专业组、同页相邻风险组和同校偏好补充组。
- `data/working/issue19-candidate-v2-major-review-seed.csv` 必须逐专业使用，判断调剂时要看该组内全部专业，不只看想填的 6 个专业。
- `C10704` 是页图可见但结构化漏拆的补种，不是已经最终核准的事实。
- `K15114` 是成都理工大学同校偏好专业补充组，含数字媒体技术，但不能继承历史候选 `K15123` 的投档线。
- `最高学费候选` 只能作为 OCR 数字线索；如果 `学费候选` 非纯数字，或命中 `tuition_not_plain_number`、`tuition_number_le_500`，预算判断必须回看原页和高校章程。
- 候选 V2 仍统一为 `candidate_v2_review_seed_needs_manual_pdf_review`，所有行 `最终可用=false`；只有完成原页、官方系统或湖北省招办计划、高校章程三方核验后，才能升级。

候选 V2 升级工作台的使用方式：

- `data/working/issue19-candidate-v2-verification-group-workbench.csv` 是专业组升级闸门表，默认 `候选闸门状态=pending_verification`、`可进入最终候选=false`。
- `data/working/issue19-candidate-v2-verification-major-workbench.csv` 是逐专业接受度和字段核验表，默认 `专业闸门状态=pending_verification`、`是否允许进入最终专业列表=false`。
- 一个专业组进入冲稳保排序前，必须补齐原 PDF 页、湖北官方系统或省招办计划、高校官网/招生章程、家庭逐专业接受度、组内调剂结论和三年投档稳定性。
- 0 明细组必须先解决 2026 组号变化或补齐专业明细；同校补充组不得沿用原历史候选投档线。

全量质量分层队列的使用方式：

- `data/working/issue19-full-quality-group-tiers.csv` 是 3329 条专业组行的质量索引，不是最终可报清单。
- `data/working/issue19-full-quality-review-queue.csv` 是优先回看原页的质量复核队列；当前 3300 条进入队列，其中 3296 条为 `P0-必须优先核页`。
- P0 触发条件包括：无专业明细、高严重结构异常、命中历史候选、命中当前偏好方向、命中硬风险、异常学费、缺再选科目、缺计划数、组表/专业明细行数不一致、规范化专业组代码重复。
- 规范化专业组代码重复不能去重处理；当前 `F01403`、`F77804`、`F79104` 各出现 2 行，必须保留原始行并回看第 19 期原页确认是否串校或切组错误。
- 后续任何候选专业组进入家庭讨论时，必须同时展开该专业组下全部专业明细；只看专业组名称、历史投档线或想填的 6 个专业，不能判断是否可服从调剂。
- `Q4-结构相对完整待抽检` 也仍是待核状态；只有完成原页、官方系统或省招办计划、高校章程和家庭接受度核验后，才可升级。

## 四、第三方数据核验

千问高考、static-data.gaokao.cn、阳光高考页面和其他工具只能作为辅助来源。

- static-data.gaokao.cn 可用于等位分交叉校验，但 2026 位次最终仍回到湖北官方一分一段。
- 千问高考可用于专业分类和候选发现，但不能直接采用其推荐概率、默认排序或预测结论。
- 阳光高考招生章程入口用于查找章程，章程内容还要逐校核对当年版本。
- 第三方来源发现的任何院校专业组，进入最终表前都必须回到湖北官方 2026 招生计划、官方投档线原件和高校招生章程复核。

## 五、提交前核验

正式提交前逐项确认：

- 登录平台网址和账号密码正确。
- 联系电话、录取通知书地址等关键信息有效。
- 45 个院校专业组顺序与最终表一致。
- 每个专业组内专业顺序一致。
- 是否服从调剂与最终表一致。
- 不愿意去的院校专业组没有被误放在前面。
- 保存、提交、确认完成，并保留最终截图或导出件到本地私密目录，不提交 Git。
