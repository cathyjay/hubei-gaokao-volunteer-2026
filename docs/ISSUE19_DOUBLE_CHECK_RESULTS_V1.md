# 第 19 期高优先级样本 Double Check 结果 V1

最后更新：2026-06-26

## 一、结论

本轮只完成“定位与风险评估”，不能把 OCR 结果直接当成招生计划最终数据。

当前第 19 期 OCR 样本足以支持：

- 定位学校所在 PDF 页码；
- 发现院校专业组大致范围；
- 生成逐页、逐组、逐字段的人工复核任务；
- 判断哪些学校官网可以作为交叉校验来源。

但当前 OCR 样本还不足以支持：

- 直接全量结构化入库；
- 直接确认院校专业组、专业代号、计划数、学费、备注；
- 直接判断某个院校专业组是否可以服从调剂；
- 直接进入最终志愿方案。

因此，本阶段所有第 19 期数据状态仍为：

```text
ocr_raw / needs_manual_review
```

不得标记为：

```text
final_allowed
```

## 四、字段可信度

相对可信：

- 学校名称命中；
- PDF OCR 页码；
- 左右栏位置；
- 院校专业组的大致出现顺序。

必须人工核验：

- 院校代码；
- 院校专业组代码；
- 专业代号；
- 专业名称；
- 专业计划数；
- 学费；
- 再选科目要求；
- 备注和特殊要求。

最高风险字段：

- `C / O / 0 / I / 1 / l` 混淆；
- 两位专业代号；
- 计划人数和学费数字；
- 中外合作办学、校区、语种、单科、色盲色弱、不予录取、体检限制、民族班、预科、专项等备注。

## 五、当前脚本边界

`scripts/extract_issue19_school_samples.py` 的当前逻辑是：

```text
学校名命中 + 前后 12 行上下文
```

这适合快速定位，不适合完整结构化。原因：

- 上下文窗口可能漏掉同一院校专业组的部分专业；
- 同页相邻学校可能混入同一个学校的上下文；
- 学校名出现在备注中时可能误召；
- 行类型里很多“专业组标题”会被归到 `school`，不能只靠行类型判断边界。

因此，样本 CSV 只能用于生成复核任务，不能直接形成正式招生计划表。

## 六、进入全量结构化前的质量门槛

高优先级 7 校必须先做到：

1. 100% 回看第 19 期原 PDF 页。
2. 逐个院校专业组确认组边界。
3. 逐组核对组内全部专业，而不是只核想填的 6 个专业。
4. 逐字段核对院校代码、专业组代码、专业代号、专业名称、计划数、学费、再选科目和备注。
5. 含 P0 问题、低置信度代码/数字、风险备注词的行必须人工复核。
6. 与学校官网、湖北官方平台或志愿系统交叉确认后，才能提升状态。

20 校样本扩展门槛：

- 高优先级 7 校达到“组边界完整、组内专业数一致、关键数字零未决、备注限制零未决”；
- 其余样本至少抽检所有命中页；
- 每校不少于 30% 专业组人工复核；
- 所有含风险备注词的专业组必须复核。

## 七、下一步

1. 第一批先做武汉科技大学、湖北大学、湖北理工学院、武汉商学院的结构化试跑。
2. 试跑只进入私有逐组复核表，不进入最终公开候选表。
3. 每个专业组必须回看第 19 期原 PDF 页，确认组边界和组内全部专业。
4. 对武汉科技大学、湖北大学、湖北理工学院，用官网专业组/查询指引做字段级交叉核验。
5. 对武汉商学院，用官网分专业计划核专业名称、湖北计划数、学费和选科；专业组代码仍回第 19 期。
6. 对湖北科技学院官网 PDF 附件，后续用长时间续传或浏览器下载补齐附件，再 OCR/人工核。
7. 样本通过后，再建立全量“院校专业组 -> 组内全部专业”结构化流程。

最终志愿表只允许使用经过人工原页复核、官方来源交叉核验、状态达到 `final_allowed` 的院校专业组。
